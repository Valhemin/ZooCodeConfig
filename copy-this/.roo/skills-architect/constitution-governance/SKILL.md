---
name: constitution-governance
description: Create or update project principles/constitution and propagate changes to templates, specs, plans, and tasks.
---

TRIGGER: project needs stable non-negotiable principles, Definition of Done, architecture constraints, or quality bars
OUTPUT: detailed — principles document with MUST/SHOULD, version, date, amendment process, compliance expectations
SKIP: feature-level decisions that don't warrant project-wide governance

---


Use when the project needs stable non-negotiable principles, Definition of Done, architecture constraints, quality bars, or governance.

Guidelines:
- Principles must be declarative, testable, and free of vague wording.
- Use MUST/SHOULD intentionally and explain rationale when not obvious.
- Record version, date, amendment process, and compliance expectations.
- After changing principles, check templates/specs/plans/tasks for alignment.
- Constitution conflicts in later analysis are critical unless the user explicitly changes the constitution.
- Leave no unexplained placeholder tokens.
